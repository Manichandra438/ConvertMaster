## Key decisions made this session
- Adopted **Vitest** as the test runner (`npm run test`) — first test suite the repo has ever had. Chose it over alternatives because the project already uses Vite; zero new build-tool surface.
- Split the single deploy workflow into two jobs: `lint-and-test` (runs on every push/PR) gating `build-and-deploy` (`needs: lint-and-test`, still `workflow_dispatch`-only for the actual Pages deploy) — kept manual-deploy design intact while adding a real CI gate that didn't exist before.
- Extracted pure logic out of components specifically to make it unit-testable (`getMimeTypeFromBytes` → `lib/mime.js`, JWT part decode → `lib/jwt.js`, resize dimension math → `lib/resize.js`) rather than testing through component rendering — kept as small, targeted extractions, not a broader refactor.

## What AI learned that isn't in docs yet
- The `handleCopy` clipboard function and the `sleep`/`formatBytes`/`triggerDownload` trio were each duplicated verbatim across 5 and 3 files respectively — both now live in `src/lib/useCopy.js` (a hook) and `src/lib/fileHelpers.js`. **This supersedes the anti-pattern logged in the earlier session today** — see anti-patterns.md update.
- Repo's ESLint config has a real gap: `no-unused-vars` doesn't know JSX counts as usage (no `eslint-plugin-react`/`jsx-uses-vars`), so every file importing `motion` from framer-motion and only using it as `<motion.div>` gets a false-positive "unused var" error. This is why lint was already broken (24 errors) before any of today's work — confirmed via `git stash` that the baseline was red, not something introduced this session. Not fixed (out of approved scope) — CI's new lint gate (M4) will fail until someone addresses this.
- File-upload dropzones across the app (`ft-dropzone` divs with `onClick`) were built as plain non-interactive `<div>`s with zero keyboard path — a pattern to watch for in any *new* file-upload UI: always pair the dropzone with `role="button" tabIndex={0}` + Enter/Space `onKeyDown`, not just `onClick`.
- Icon-only buttons throughout the tool pages relied on `title` alone; some (PdfTool's file-list Download/Remove, Base64Tool's file-remove) had **no accessible name at all**. New icon-only buttons need `aria-label` from the start.
- Browser-automation gotcha (now documented in the new `.claude/skills/verify/SKILL.md`): the `computer` tool's `type` action silently drops characters on long strings; CDP screenshots intermittently time out right after CPU-heavy work (pdf-lib) but succeed on retry; framer-motion fade-ins can make a fresh screenshot look blank if taken too fast.
- Saw a one-off "Maximum update depth exceeded" React crash on the JWT page during a heavy scripted-interaction burst; could not reproduce on a clean retry, and the stack trace's cited line didn't match real source (pointed at a framer-motion prop, not an event handler) — most likely a stale dev-server/HMR artifact from a long live-edit session, not a shipped bug. Flagged, not confirmed.
